# ТЗ: Redmine Slow Query Logger

## 1. Назначение

Плагин предназначен для выявления пользователей, фильтров, URL и SQL-запросов, которые создают высокую нагрузку на Redmine 6.0.x и базу данных.

Основная задача: дать администратору Redmine понятный журнал, где видно не только сам тяжелый SQL, но и контекст:

- кто инициировал запрос;
- какой URL/раздел Redmine был открыт;
- сколько занял web-запрос;
- сколько SQL было выполнено внутри запроса;
- какой SQL был медленным;
- с какого IP и request id пришел запрос;
- насколько событие превышает заданный порог.

Плагин не должен заменять APM или slow query log БД. Он закрывает прикладной уровень Redmine: связь “пользователь -> страница Redmine -> SQL”.

## 2. Целевая среда

- Redmine: 6.0.x, первичная версия проверки: 6.0.5.
- Rails: версия, поставляемая с Redmine 6.0.x.
- Ruby: версия, поддерживаемая Redmine 6.0.x.
- БД: PostgreSQL как основной сценарий, совместимость с MySQL/MariaDB не должна намеренно ломаться.
- Режим эксплуатации: production.

## 3. Роли пользователей

### Администратор Redmine

Основной пользователь плагина.

Задачи:

- увидеть, кто создает тяжелую нагрузку;
- отфильтровать журнал по пользователю, типу события, времени;
- открыть полный SQL;
- понять, какой URL или фильтр задач вызвал нагрузку;
- очистить журнал;
- настроить пороги и глубину хранения.

### Системный администратор / DevOps

Задачи:

- включить плагин в production;
- настроить пороги через ENV или настройки Redmine;
- сопоставить события плагина с `production.log`, nginx access log, PostgreSQL slow log;
- проверить, не создает ли сам плагин дополнительную существенную нагрузку.

### Аналитик / Руководитель проекта

Ограниченный сценарий: может получать выгрузку или скриншот журнала от администратора.

Не должен иметь прямой доступ к SQL, если это раскрывает чувствительные данные.

## 4. Основные сценарии

### 4.1. Найти пользователя, который создает тяжелые запросы

1. Администратор открывает `Администрирование -> Slow query log`.
2. Видит последние события, отсортированные от новых к старым.
3. Фильтрует по типу `Request` или `SQL`.
4. Смотрит `login`, `user_id`, IP, URL, длительность.
5. По URL понимает, какой раздел Redmine вызвал нагрузку.

### 4.2. Проверить тяжелый фильтр задач

1. Пользователь открывает `/issues?...`.
2. Redmine выполняет тяжелый `COUNT(*)` или выборку задач.
3. Плагин фиксирует:
   - событие `request`;
   - одно или несколько событий `sql`;
   - путь `/issues?...`;
   - `User.current`;
   - суммарное время SQL.
4. Администратор видит, что проблема связана с конкретным фильтром.

### 4.3. Проверить работу плагина без реальной нагрузки

1. Администратор запускает rake-задачу генерации нагрузки.
2. Задает `USER_ID`, `ITERATIONS`, `PROJECT_LIMIT`.
3. Плагин пишет тестовые события в журнал.
4. Администратор проверяет UI и `production.log`.

## 5. Функциональные требования

### 5.1. Логирование медленных SQL

Плагин должен подписываться на `sql.active_record`.

Для каждого SQL дольше порога фиксировать:

- тип события: `sql`;
- дата и время;
- длительность в мс;
- порог в мс;
- `user_id`;
- `login`;
- `request_id`;
- IP;
- HTTP-метод;
- URL/path;
- имя SQL-события ActiveRecord;
- SQL-текст, нормализованный по пробелам;
- признак обрезки SQL, если текст превышает лимит.

Плагин не должен логировать собственные INSERT/DELETE в таблицу журнала, чтобы не создавать рекурсию.

### 5.2. Логирование медленных web-запросов

Плагин должен оборачивать Rails request middleware.

Для каждого web-запроса дольше порога фиксировать:

- тип события: `request`;
- дата и время;
- длительность в мс;
- HTTP-статус;
- `user_id`;
- `login`;
- `request_id`;
- IP;
- HTTP-метод;
- URL/path;
- количество SQL внутри web-запроса;
- количество медленных SQL внутри web-запроса;
- суммарное время SQL.

### 5.3. Журнал в интерфейсе Redmine

Плагин должен добавить страницу журнала в административный раздел.

Минимальные элементы:

- таблица событий;
- сортировка по времени от новых к старым;
- фильтр по типу события: все / SQL / Request;
- фильтр по логину;
- лимит отображения;
- кнопка очистки журнала;
- ссылка на настройки плагина;
- безопасный HTML-escape SQL, URL и пользовательских данных.

### 5.4. Настройки

В настройках плагина должны быть:

- включение/отключение записи в DB-журнал;
- максимальное количество хранимых записей;
- ссылка на журнал.

Через ENV должны поддерживаться:

- `REDMINE_SLOW_SQL_MS`;
- `REDMINE_SLOW_REQUEST_MS`;
- `REDMINE_SLOW_SQL_MAX_LENGTH`;
- `REDMINE_SLOW_SQL_LOG_ALL`;
- `REDMINE_SLOW_REQUEST_LOG_ALL`;
- `REDMINE_SLOW_LOG_MAX_ENTRIES`.

Перспективное улучшение: перенести пороги из ENV в UI-настройки, оставив ENV как override.

### 5.5. Генерация тестовой нагрузки

Плагин должен иметь rake-задачу:

```bash
bundle exec rake redmine_slow_query_logger:db_load RAILS_ENV=production
```

Параметры:

- `USER_ID` - от имени какого пользователя установить `User.current`;
- `ITERATIONS` - количество повторов;
- `PROJECT_LIMIT` - сколько проектов включать в тестовый `IN (...)`.

Задача должна выполнять запрос, похожий на тяжелый список задач Redmine:

- `Issue`;
- `Project`;
- `IssueStatus`;
- проверка `enabled_modules`;
- `COUNT(*)`;
- список проектов.

## 6. Нефункциональные требования

### 6.1. Производительность

Плагин сам не должен становиться источником существенной нагрузки.

Требования:

- не выполнять тяжелые вычисления на каждый SQL;
- не строить stacktrace в production;
- не сохранять SQL меньше порога, если не включен debug-режим;
- обрезать длинный SQL;
- автоматически удалять старые записи;
- не логировать собственные операции записи журнала.

Рекомендуемые production-пороги:

- SQL: 500-1000 мс;
- web-запрос: 1000-3000 мс;
- максимум записей: 1000-10000, в зависимости от нагрузки.

### 6.2. Достоверность

Плагин должен явно разделять факты и ограничения.

Фактически достоверно:

- какой `User.current` был в момент обработки web-запроса;
- какой URL обрабатывал Redmine;
- какой SQL прошел через ActiveRecord;
- сколько времени заняло выполнение по измерению Rails.

Ограничения достоверности:

- если запрос идет из фоновой задачи, `User.current` может быть anonymous или nil;
- если SQL выполняется вне web-request контекста, IP/URL/request id могут отсутствовать;
- если пользователь работает через общий аккаунт, плагин покажет общий аккаунт;
- если используется API key, нужно отдельно проверить, устанавливает ли Redmine `User.current`;
- длительность SQL по Rails может отличаться от длительности в PostgreSQL slow log;
- плагин показывает инициатора на уровне Redmine, но не заменяет аудит на уровне reverse proxy и БД.

Журнал должен помогать расследованию, но не быть единственным источником для юридически значимого аудита.

### 6.3. Безопасность

SQL и URL могут содержать чувствительные данные.

Требования:

- доступ к журналу только администраторам;
- HTML-escape всех выводимых значений;
- ограничение длины SQL;
- возможность очистить журнал;
- возможность отключить DB-журнал;
- не показывать журнал обычным пользователям;
- не писать пароли или секреты намеренно.

Перспективное улучшение:

- маскирование параметров URL;
- маскирование SQL literals;
- настройка “не хранить полный SQL, только начало/statement type/table names”.

### 6.4. Удобство

Журнал должен отвечать на вопрос администратора за 10-20 секунд:

- кто;
- когда;
- что открыл;
- насколько долго;
- какой SQL был самым тяжелым.

Таблица должна быть читаемой без просмотра `production.log`.

Важные UX-решения:

- request-события показывать как сводку;
- SQL-события показывать с раскрытым SQL;
- длинный SQL выводить в `<pre>` с переносами;
- новые записи сверху;
- фильтры над таблицей;
- явное отображение порога.

## 7. Модель данных

Таблица: `slow_query_logger_entries`.

Поля:

- `id`;
- `event_type`;
- `duration_ms`;
- `threshold_ms`;
- `status`;
- `user_id`;
- `login`;
- `request_id`;
- `ip`;
- `http_method`;
- `path`;
- `sql_name`;
- `sql_count`;
- `slow_sql_count`;
- `sql_duration_ms`;
- `sql`;
- `created_at`;
- `updated_at`.

Индексы:

- `created_at`;
- `event_type`;
- `login`;
- `user_id`.

## 8. Интеграция с внешними логами

Каждая запись должна содержать `request_id`, если он доступен.

Это позволит сопоставлять:

- UI-журнал плагина;
- `log/production.log`;
- nginx/apache access log;
- PostgreSQL slow query log;
- APM-трассировки.

## 9. Критерии приемки

### 9.1. Установка

- Плагин копируется в `plugins/redmine_slow_query_logger`.
- `bundle exec rake redmine:plugins:migrate RAILS_ENV=production` создает таблицу.
- После рестарта Redmine плагин виден в `Администрирование -> Плагины`.
- Настройки плагина открываются без ошибки.

### 9.2. Логирование SQL

При `REDMINE_SLOW_SQL_MS=0` любое SQL-событие из тестовой rake-задачи появляется:

- в `production.log`;
- в UI-журнале.

Запись содержит:

- тип `sql`;
- длительность;
- пользователя, если задан `USER_ID`;
- SQL-текст.

### 9.3. Логирование request

При `REDMINE_SLOW_REQUEST_MS=0` открытие любой страницы Redmine создает `request`-событие.

Запись содержит:

- URL;
- HTTP-метод;
- статус;
- пользователя;
- количество SQL;
- суммарное SQL-время.

### 9.4. UI

- Администратор видит журнал.
- Обычный пользователь не имеет доступа.
- Фильтр по типу события работает.
- Фильтр по логину работает.
- Очистка журнала удаляет все записи.
- HTML/SQL в журнале не ломает страницу.

### 9.5. Ограничение хранения

Если максимум записей равен 1000, после превышения лимита старые записи удаляются.

## 10. Риски

### Риск: журнал сам создает нагрузку

Меры:

- хранить только медленные события;
- лимитировать количество записей;
- отключать DB-журнал;
- обрезать SQL;
- не логировать собственные INSERT/DELETE.

### Риск: утечка чувствительных данных через SQL

Меры:

- доступ только администраторам;
- ограничение длины SQL;
- возможность очистки;
- будущая маскировка literals и URL-параметров.

### Риск: неверная атрибуция пользователя

Меры:

- явно хранить `User.current`;
- хранить IP и request id;
- в документации указать ограничения;
- для API и фоновых задач проверять отдельно.

### Риск: несовместимость с будущими Redmine/Rails

Меры:

- минимально использовать публичные Rails API;
- не патчить Redmine core;
- покрыть smoke-тестами установку, миграцию, открытие UI.

## 11. Дальнейшие улучшения

- UI-настройки порогов SQL/request.
- Фильтр по диапазону дат.
- Фильтр по IP.
- Фильтр “только события дольше N мс”.
- Экспорт CSV.
- Маскирование SQL literals.
- Группировка одинаковых SQL по fingerprint.
- Страница “топ пользователей по нагрузке”.
- Страница “топ URL по нагрузке”.
- Автоматическая привязка SQL-событий к request-событию.
- Поддержка фоновых задач с отдельным контекстом.
- Интеграция с PostgreSQL `pg_stat_statements`.

## 12. Минимальная версия MVP

MVP считается достаточным, если:

- есть логирование медленных SQL;
- есть логирование медленных web-запросов;
- есть DB-журнал в админке;
- есть фильтры тип/логин/лимит;
- есть очистка журнала;
- есть rake-задача генерации нагрузки;
- есть защита от рекурсивного логирования собственных записей;
- доступ только администраторам.
